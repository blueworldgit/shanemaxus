1. Project Overview

The goal was to transform the homepage and navigation of shane.maxusvanparts.co.uk to match the design and functionality of maxusvanparts.acstestweb.co.uk (the source/reference site), while keeping the existing WooCommerce database structure intact.

What was built

  • Homepage visual facelift (hero banner, vehicle carousel, Why Us section, custom footer)
  • Vehicle landing pages at /vehicle/{slug}/ with organic DB-driven category grids
  • Department pages at /department/{slug}/ aggregating vehicles by part category
  • VIN lookup (17-character VIN → vehicle identification)
  • Registration lookup (UK plate → vehicle identification via external API)
  • Replacement search bar (Model / Year / VIN / Registration)
  • Header dropdown lookup panels on nav menu items

What was NOT changed

  • No WooCommerce products, categories, or core data were modified
  • No existing database tables or structures were altered
  • No plugins were added or removed
  • The existing Mobex parent theme was not touched

Section 5: Vehicle Landing Pages — How It Works

  1. mvp_vehicle_rewrite_rules() — Registers URL pattern ^vehicle/([^/]+)/?$
  2. mvp_vehicle_query_vars() — Registers mvp_vehicle query var
  3. mvp_vehicle_prevent_404() — Prevents WP from setting 404 on vehicle pages
  4. mvp_vehicle_template_redirect() — Looks up VIN term by vehicle_slug meta, loads child categories
  5. mvp_vehicle_render_full_page() — Outputs get_header(), custom HTML/CSS, get_footer()

Category images use convention: https://maxusvanparts.acstestweb.co.uk/wp-content/uploads/categories/{Category_Name}.png (spaces replaced with underscores)

Section 7: VIN Lookup — Matching Logic

  1. Validates: 17 characters, starts with “LS” (Maxus manufacturer code)
  2. Exact match — full VIN matches a known VIN serial exactly
  3. Pattern match — first 8 characters (vehicle pattern), finds all VINs with same prefix
  4. Year narrowing — VIN position 10 = model year (A=2010, B=2011, … M=2021, N=2022, … S=2025)

Section 8: Registration Lookup — Flow

  1. Calls checkcardetails.co.uk API with registration plate
  2. Extracts make, model, year, fuel type from response
  3. Validates it’s a Maxus/LDV/SAIC/MG vehicle
  4. Matches model name to vehicle database using keyword matching
  5. Handles electric vs diesel distinction (fuel type “Electricity” → E Deliver variants)

8. Known Issues & Limitations

8a. Miscategorized Products Pre-existing Data Issue

Some products are assigned to incorrect categories in the database. For example, “BALL ASSEMBLY-FRONT SUSPENSION UPPER CONTROL ARM” (a suspension part) is categorised under Brakes > Front Brakes for several VINs but correctly under Suspension > Front Suspension for others.

Affected VINs (wrong category): LSH14J7CXMA114599, LSH14J7C9RV123360, LSH14J7C4RV123458
Correct on: LSFAM11C6RA144501

This is an upstream data import issue. Our code filters correctly by category — the category assignments themselves are wrong for some products.

8c. Duplicate SKUs Across VINs Scaling Concern

The same physical part (same SKU) exists as separate WooCommerce products under different VIN categories. This creates:

  • Scale concern: Adding another brand (e.g. MG) will multiply product count significantly
  • Google SEO concern: SKU searches could land on any VIN’s version. The page needs to show vehicle compatibility.
Recommendation: Consider whether a single-product-per-SKU approach with vehicle compatibility attributes would be more sustainable long-term, especially for multi-brand scaling.
